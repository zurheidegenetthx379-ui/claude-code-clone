# 第八章：Prompt 管理机制与实现细节

## 8.1 六层 Prompt 运行时系统

Claude Code 的 Prompt 管理并非简单的字符串拼接，而是一个精心分层的运行时系统。从底到顶，六个层次各司其职：

| 层级 | 名称 | 职责 | 变更频率 |
|------|------|------|---------|
| L1 | 系统基础层 | 模型行为约束、安全边界 | 极低（版本级） |
| L2 | 工具定义层 | 所有可用工具的 JSON Schema 声明 | 低（配置级） |
| L3 | 上下文注入层 | CLAUDE.md、Git 状态、日期 | 中（会话级） |
| L4 | 技能注入层 | 已注册技能的描述和触发规则 | 中（项目级） |
| L5 | 对话历史层 | Transcript 消息序列 | 高（轮次级） |
| L6 | 动态指令层 | 运行时追加的特殊指令 | 高（操作级） |

这种分层设计的核心好处是**缓存友好性**：低层内容变化频率低，可以长时间命中 API 的 prompt cache，显著降低延迟和成本。

## 8.2 默认提示词的分段构建

默认系统提示词的构建采用"静态段 + 动态边界 + 动态段"的三段式结构：

```
┌──────────────────────────────────┐
│ 静态段（Static Section）         │  ← 模型身份、行为规则、工具使用指南
│ 极少变化，高度可缓存             │
├──────────────────────────────────┤
│ SYSTEM_PROMPT_DYNAMIC_BOUNDARY   │  ← 哨兵值标记
├──────────────────────────────────┤
│ 动态段（Dynamic Section）        │  ← 当前日期、项目上下文、Git 状态
│ 每次会话可能变化                 │
└──────────────────────────────────┘
```

**静态段**包含约 4000 token 的固定内容，定义了 Claude Code 的核心行为模式：如何与用户交互、何时使用工具、如何处理错误。这部分内容在所有用户、所有项目、所有会话中完全相同，是 prompt cache 命中率的最大贡献者。

**动态段**包含运行时特定的上下文信息，如当前日期、工作目录路径、活跃的 Git 分支名称、最近的文件变更摘要等。

## 8.3 优先级装配机制

当存在多个提示词来源时，系统按以下优先级顺序装配：

```
override > coordinator > agent > custom > default + append
```

- **default**：默认系统提示词，作为基础层。
- **custom**：用户通过 `CLAUDE.md` 或 `settings.json` 中的 `customInstructions` 提供的指令。以 `append` 方式追加在 default 之后。
- **agent**：当代理被显式指定角色（如 "你是一个专注于安全的审查者"）时使用的提示词。替换 default 中的角色定义段。
- **coordinator**：多代理协调模式下，协调者角色的专用提示词。完全替换 default。
- **override**：最高优先级，由内部系统在某些特殊场景下使用（如安全审计模式），完全覆盖所有其他来源。

装配逻辑的关键设计是**粒度控制**：`custom` 和 `append` 是追加模式（保留 default），而 `agent`、`coordinator` 和 `override` 是替换模式（丢弃 default）。这种混合策略既允许用户增量定制，也允许系统级完全接管。

## 8.4 运行时上下文注入

系统提示词的动态段包含多种运行时上下文信息：

**CLAUDE.md 注入**：系统会递归查找项目目录树中的所有 `CLAUDE.md` 文件（从项目根目录到当前工作目录），将内容合并后注入。合并顺序是从根到叶，允许子目录的 `CLAUDE.md` 覆盖或补充上级目录的指令。

**日期注入**：当前日期以 `Today's date is YYYY-MM-DD` 格式注入，使模型能够理解时间相关的请求（如"上周的代码变更"）。

**Git 状态注入**：当前分支名称、最近的 commit hash、工作区状态（是否有未提交的变更）会被注入。这使得模型在执行 Git 相关操作时具备项目状态感知能力。

```typescript
interface RuntimeContext {
  claudeMdContent: string;    // 合并后的 CLAUDE.md 内容
  currentDate: string;        // ISO 8601 日期
  gitBranch?: string;         // 当前 Git 分支
  gitStatus?: string;         // 工作区状态摘要
  activeSkills: string[];     // 当前可用的技能列表
}
```

## 8.5 分段缓存与失效策略

缓存策略的核心是将系统提示词分为两个缓存域：

**`systemPromptSection`（可缓存域）**：包含静态段内容。这部分被标记为可缓存，API 调用时会利用 Anthropic 的 prompt caching 机制，避免重复处理不变的提示词前缀。缓存的 key 基于内容的哈希值，确保内容变更时缓存自动失效。

**`DANGEROUS_uncached`（不可缓存域）**：包含动态段和所有后续内容（工具定义、对话历史）。以 `DANGEROUS_` 前缀命名是一种防御性编程实践——提醒开发者这个域中的内容每次都需要重新处理，不能假设缓存命中。

```typescript
const promptPayload = {
  system: [
    { type: "text", text: staticSection, cache_control: { type: "ephemeral" } },
    { type: "text", text: SYSTEM_PROMPT_DYNAMIC_BOUNDARY },
    { type: "text", text: dynamicSection },  // DANGEROUS_uncached
  ],
  // ...
};
```

## 8.6 动态边界哨兵值

`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 是一个特殊的哨兵值（sentinel value），它的作用是在静态段和动态段之间建立一个明确的分界标记。

为什么需要这个分界？因为 prompt caching 的工作机制是**前缀匹配**——从提示词开头开始，逐 token 匹配缓存，直到遇到第一个不匹配的 token 为止。哨兵值确保了：

1. 静态段的缓存不会"泄漏"到动态段——即使动态段开头的内容碰巧与某次历史调用的动态段相同。
2. 监控系统可以通过检测哨兵值的位置来验证缓存边界是否正确设置。
3. 在调试模式下，哨兵值会被替换为可辨识的标记字符串，方便开发者在 API 请求日志中定位缓存边界。

## 8.7 任务专用提示词族

除了通用系统提示词，系统还维护了一组**任务专用提示词**，用于特定内部流程：

**Compact 提示词**：用于对话历史压缩。当对话长度接近上下文窗口限制时，系统会调用一个专用的压缩 LLM 调用，使用此提示词指导模型将冗长的对话历史浓缩为关键信息摘要。

```
请将以下对话历史压缩为简洁的摘要，保留：
1. 用户的核心需求和意图
2. 已完成的文件操作和代码变更
3. 未解决的问题和待执行的步骤
4. 重要的上下文信息（项目结构、技术栈）
```

**Session Memory 提示词**：用于会话记忆提取。在会话结束时，系统会分析整个对话，提取值得长期记忆的信息（如用户偏好、项目约定），存储到 memory 系统中供后续会话使用。

**Extract Memories 提示词**：用于从压缩后的对话中进一步蒸馏记忆。这是 Session Memory 的精简版，在 compact 操作后自动触发，确保即使被压缩过的对话也不会丢失关键记忆。

这三个提示词形成了一个**信息蒸馏链**：完整对话 → Compact（保留关键信息） → Session Memory（提取长期记忆） → Extract Memories（从压缩内容中二次提取）。每一层都在减少信息量的同时保留最有价值的部分。
