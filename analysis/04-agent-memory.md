# 第三章：Agent Memory 机制

## 3.1 四层记忆架构

Claude Code 的记忆系统分为四个层次，每层有不同的生命周期、作用域和持久化方式：

```
┌────────────────────────────────────────────────────────────────────┐
│                                                                    │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │            Layer 4: Team Memory (团队记忆)                    │  │
│  │    作用域：团队/组织    存储：共享 Git 仓库                     │  │
│  │    生命周期：永久       维护：人工审核                           │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                    │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │          Layer 3: Agent Memory (Agent 记忆)                   │  │
│  │    作用域：Agent 角色   存储：MEMORY.md 快照                   │  │
│  │    生命周期：跨项目     维护：手动创建/更新                      │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                    │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │         Layer 2: Session Memory (会话记忆)                    │  │
│  │    作用域：当前会话    存储：.claude/session-memory/           │  │
│  │    生命周期：单次会话   维护：自动（子代理沙箱）                  │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                    │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │         Layer 1: Auto Memory (自动记忆)                       │  │
│  │    作用域：当前项目    存储：CLAUDE.md                          │  │
│  │    生命周期：跨会话     维护：自动追加                           │  │
│  └──────────────────────────────────────────────────────────────┘  │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

### 各层对比

| 维度 | Auto Memory | Session Memory | Agent Memory | Team Memory |
|------|------------|----------------|-------------|-------------|
| 存储位置 | `CLAUDE.md` | 临时沙箱文件 | `MEMORY.md` 快照 | 共享 Git 仓库 |
| 写入方式 | Agent 自动追加 | 子代理自动生成 | 用户手动创建 | 人工审核提交 |
| 注入时机 | 每次会话开始 | 上下文压缩后 | 角色匹配时 | 团队配置加载时 |
| 持久性 | 跨会话 | 单次会话 | 跨项目 | 永久 |
| 大小限制 | 硬截断保护 | 沙箱内限制 | 索引+按需加载 | 手动维护 |

## 3.2 Session Memory：触发阈值与沙箱化子代理

Session Memory 是四层记忆中最精妙的部分。它不是在每次对话后都更新，而是有**触发阈值**：

```
用户对话进行中...
    │
    ▼
Agent 检测到值得记忆的内容
    │
    ├─ 重要性评估
    │   ├─ 低：忽略（临时调试、一次性操作）
    │   ├─ 中：记录到工作记忆（Working Memory）
    │   └─ 高：触发 Session Memory 写入
    │
    ▼
[沙箱化子代理] 启动
    │
    ├─ 在隔离环境中运行
    ├─ 只能访问当前会话上下文
    ├─ 生成结构化的记忆片段
    └─ 写入 session-memory/ 临时目录
```

**沙箱化设计的原因**：记忆生成过程本身需要调用 LLM，如果这个过程在主 Agent 循环中同步执行，会导致：
1. 主循环被阻塞，用户等待时间增加
2. 记忆生成的 Tool Call 污染主对话的上下文
3. 记忆生成失败可能影响主循环稳定性

因此，Session Memory 的写入在一个**独立的子代理沙箱**中完成，与主 Agent 循环完全隔离。

## 3.3 上下文压缩（Compaction）机制

当对话轮次增多，上下文窗口逼近模型的 Token 上限时，Claude Code 启动**上下文压缩**：

```
对话历史（完整）
    │
    ▼
[上下文大小检测]
    │
    ├─ 未超阈值 → 继续累积
    │
    └─ 超过阈值（如 80% Token 上限）
        │
        ▼
    [Compaction 流程]
        │
        ├─ 1. 提取关键信息摘要
        │      - 已完成的任务及其结果
        │      - 未解决的问题和当前状态
        │      - 用户的偏好和约束
        │      - 重要的代码变更
        │
        ├─ 2. 丢弃冗余细节
        │      - 中间调试过程
        │      - 已修正的错误尝试
        │      - 冗长的工具输出原文
        │
        ├─ 3. 生成压缩后的上下文
        │      - 保留所有决策点和结论
        │      - 用摘要替代详细过程
        │
        └─ 4. 注入 Session Memory 片段
               - 从 session-memory/ 召回相关记忆
               - 补充压缩时可能丢失的关键信息
```

压缩的目标不是"尽可能多保留原文"，而是"保留决策链路"——让 Agent 在压缩后仍然知道自己做了什么、为什么做、还有什么没做完。

## 3.4 相关记忆召回

记忆注入采用**轻量检索**而非全量注入的策略：

```
当前用户输入: "帮我修复登录页面的样式问题"
    │
    ▼
[记忆检索模块]
    │
    ├─ 从 MEMORY.md 索引中检索
    │   → 命中: "登录页面使用 Tailwind CSS, 主题色 #3B82F6"
    │
    ├─ 从 Auto Memory (CLAUDE.md) 检索
    │   → 命中: "CSS 规范使用 BEM 命名, 全局变量在 theme.css"
    │
    └─ 从 Session Memory 检索
        → 命中: "上一轮对话中用户要求保持响应式布局"
    │
    ▼
[组装注入]
    只将相关记忆片段注入 System Prompt
    而非注入全部记忆内容
```

这避免了两个问题：
1. **Token 浪费**：全量注入会消耗大量 Token 在不相关的记忆上
2. **注意力稀释**：LLM 面对过多上下文时，对关键信息的注意力会被稀释

## 3.5 MEMORY.md 索引设计与硬截断保护

Agent Memory 的存储核心是 `MEMORY.md` 文件，它采用**索引 + 分片**的设计：

```markdown
# MEMORY.md - Agent Memory 索引

## 前端开发
- 技术栈: React 18 + TypeScript + Tailwind CSS
- 状态管理: Zustand (非 Redux)
- 详细记忆: [frontend-details.md](./memory/frontend-details.md)

## 后端开发
- 运行时: Node.js 20 + Express
- 数据库: PostgreSQL + Prisma ORM
- 详细记忆: [backend-details.md](./memory/backend-details.md)

## 部署配置
- 平台: AWS ECS + Docker
- CI/CD: GitHub Actions
- 详细记忆: [deployment-details.md](./memory/deployment-details.md)
```

**硬截断保护**：当 `MEMORY.md` 文件超过预设大小（例如 2000 行或特定 Token 数），系统会执行硬截断——超出的内容不会被加载。这防止了记忆文件无限增长导致的 Token 溢出问题。但硬截断是最后手段，正常流程中 Agent 会通过索引设计让单文件保持精简。

## 3.6 Agent Memory 快照：可分发的角色资产

Agent Memory 的一个独特用法是**快照导出**。用户可以将某个 Agent 角色的完整记忆（`MEMORY.md` + 关联的分片文件）打包为快照，分发给团队成员：

```
角色: "高级前端工程师"
    │
    ├─ MEMORY.md (索引)
    ├─ memory/
    │   ├─ frontend-details.md
    │   ├─ design-system.md
    │   └─ testing-standards.md
    │
    └─ 导出为 agent-snapshot-frontend.tar.gz
        │
        └─ 团队成员解压后加载
            → 获得该角色的完整上下文和偏好
```

这种机制让团队可以在不同成员之间**复用 Agent 的角色知识**，无需每次从头配置。

## 3.7 文件系统存储而非数据库的设计哲学

Claude Code 的四层记忆全部使用**纯文件系统存储**（Markdown 文件 + 目录结构），而非 SQLite 或其他数据库。这是刻意的设计选择：

### 为什么选择文件系统？

| 考量 | 文件系统 | 数据库 |
|------|---------|-------|
| 可审查性 | Markdown 人类可读，`cat` 即可查看 | 需要专用工具查询 |
| 版本控制 | 可直接 `git add` 纳入版本管理 | 二进制文件难以 diff |
| 可移植性 | 跨平台零依赖 | 需要数据库运行时 |
| 可编辑性 | 用户可用任何编辑器修改 | 需要通过 API 或客户端 |
| 透明度 | 所见即所得，无隐藏数据 | Schema 变更不透明 |

### 代价

- 无事务保障（并发写入可能冲突，但 Agent 单线程特性避免了此问题）
- 无索引加速（依赖线性扫描，但记忆文件通常很小）
- 无查询语言（依赖 LLM 语义理解，而非 SQL 查询）

```
~/.claude/
├── projects/
│   └── <project-hash>/
│       ├── CLAUDE.md           ← Auto Memory
│       ├── session-memory/     ← Session Memory（临时）
│       │   ├── insight-001.md
│       │   └── insight-002.md
│       └── memory/             ← Agent Memory 分片
│           ├── frontend.md
│           └── backend.md
├── MEMORY.md                   ← Agent Memory 索引
└── teams/
    └── <team-name>/            ← Team Memory
        └── shared-knowledge.md
```

---

> **小结**：Claude Code 的记忆系统是一个精心分层的架构。Auto Memory 实现零配置的项目记忆，Session Memory 通过沙箱子代理在会话内自动提炼知识，Agent Memory 提供可分发的角色快照，Team Memory 支撑团队级知识共享。上下文压缩和相关记忆召回确保 Token 使用效率，而文件系统存储的选择体现了"透明优先"的设计哲学。
