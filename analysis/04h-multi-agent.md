# 第九章：Multi-Agent 机制与实现细节

## 9.1 三种并发多代理模型

Claude Code 的多代理系统提供三种递进的并发模型，适用于不同复杂度的任务场景：

| 模型 | 关系 | 通信方式 | 典型场景 |
|------|------|---------|---------|
| 普通子代理 | 父子（单向控制） | 结果回传 | 独立子任务委托 |
| 协调者模式 | 领导-下属（双向） | 文件邮箱 + 任务恢复 | 多步骤协调任务 |
| Swarm 团队 | 对等协作 | 共享文件系统 | 并行探索与竞争 |

三种模型共享同一套底层基础设施（进程管理、沙箱隔离、权限控制），但在编排逻辑和通信机制上各有不同。

## 9.2 AgentTool 统一入口与路由逻辑

所有多代理交互都通过 `AgentTool` 这个统一入口进行路由。当主代理（或用户）请求创建一个子代理时，`AgentTool` 负责：

1. **参数校验**：验证子代理的类型、权限范围、资源配额是否合法。
2. **路由分发**：根据请求的 `mode` 参数（`subagent` / `coordinator` / `swarm`）将控制权转交给对应的编排器。
3. **生命周期管理**：跟踪子代理的创建、运行、完成/失败状态，提供统一的查询接口。

```typescript
class AgentTool {
  async execute(params: AgentParams): Promise<AgentResult> {
    const validated = this.validateParams(params);
    const orchestrator = this.route(validated.mode);
    const agent = await orchestrator.create(validated);
    return agent.run();
  }
}
```

路由逻辑的设计遵循"开闭原则"：添加新的代理模型只需实现一个新的 `Orchestrator` 接口并注册到路由表，无需修改 `AgentTool` 的核心代码。

## 9.3 普通子代理执行链

普通子代理的执行遵循一条严格的四阶段链：

```
MCP 初始化 → 上下文构建 → Hook 执行 → Query 调用
```

**阶段一：MCP 初始化。** 子代理继承父代理的 MCP 连接池，但建立独立的会话上下文。这意味着子代理可以使用父代理已建立的 MCP 服务器连接，但其工具调用不会影响父代理的 MCP 状态。

**阶段二：上下文构建。** 子代理的上下文从精简的默认提示词开始，然后根据任务描述注入特定的指令。关键点在于：子代理**不继承**父代理的完整对话历史，而是只接收一个任务摘要。这种设计控制了子代理的上下文窗口使用量，也避免了历史信息对子代理决策的干扰。

**阶段三：Hook 执行。** 如果系统中注册了 `beforeSubagentRun` 类型的 Hook，它们会在此阶段被调用。Hook 可以修改子代理的配置（如收紧权限）、注入额外上下文、甚至直接拒绝子代理的创建。

**阶段四：Query 调用。** 子代理向 LLM API 发送请求并等待响应。响应结果经过后处理后返回给父代理。

## 9.4 Fork 子代理变体

Fork 子代理是普通子代理的一个特殊变体，其核心差异在于**缓存优化策略**。

在普通子代理模式下，子代理使用独立构建的提示词，与父代理的提示词前缀不同，因此无法共享 prompt cache。而 Fork 子代理通过**继承父代理的渲染提示词字节**（rendered prompt bytes）来确保缓存命中：

```typescript
class ForkSubAgent extends SubAgent {
  constructor(parent: Agent, task: string) {
    // 复制父代理的完整渲染提示词作为前缀
    const inheritedPrompt = parent.getRenderedPromptBytes();
    // 在继承的提示词后追加 Fork 任务指令
    super({ prompt: inheritedPrompt + forkTaskSuffix(task) });
  }
}
```

这种设计在以下场景中特别有效：当需要基于当前对话上下文执行一个"分叉"任务（如"在不修改当前上下文的情况下探索另一种实现方案"）时，Fork 子代理可以复用父代理已经缓存的大量 prompt 前缀，显著降低首次 token 处理延迟。

## 9.5 协调者模式的身份重写

协调者模式引入了一个独特的**身份重写**（Identity Rewriting）机制。当协调者代理创建下属代理时，它会为每个下属代理生成一个定制化的身份描述，注入到该下属代理的系统提示词中：

```
你是协调者团队的成员之一，专长领域：前端组件测试。
你的任务由团队协调者分配。完成后将结果通过文件邮箱返回给协调者。
团队成员：[Alice-后端架构, Bob-前端测试, Charlie-性能优化]
```

身份重写的目的是让每个下属代理明确自己在团队中的角色定位，避免多个下属代理产生重复工作或角色冲突。协调者通过分析任务需求，动态决定需要哪些角色，并为每个角色生成最匹配的身份描述。

## 9.6 Swarm 的团队创建与队友生成

Swarm 模式是最高级的多代理模型，支持**动态团队创建**：

**团队创建**：一个"领导者"代理发起团队创建请求，指定团队规模和协作目标。系统创建对应数量的代理实例，每个实例具有独立的上下文和工具访问权限。

**队友生成**：领导者可以为每个队友指定不同的"策略倾向"（如"激进探索"、"保守验证"、"全面审查"），使得团队内部形成多样化的问题解决策略。

**进程内队友**：Swarm 支持一种轻量级的"进程内队友"变体，队友不启动独立的 API 请求循环，而是在领导者的 API 调用中以 `tool_use` 的形式被调用。这种模式减少了并发 API 请求数，适用于 API 速率受限的场景。

## 9.7 双轨通信机制

多代理间的通信采用**双轨制**：

**轨道一：文件邮箱。** 每个代理拥有一个以代理 ID 命名的邮箱文件（位于临时目录中）。代理间通过向对方的邮箱文件写入消息来实现异步通信。文件邮箱的优势是持久化——即使接收方代理暂时繁忙，消息也不会丢失。

```typescript
interface MailboxMessage {
  from: string;       // 发送方代理 ID
  to: string;         // 接收方代理 ID
  type: "task" | "result" | "status";
  content: string;    // 消息内容
  timestamp: number;
}
```

**轨道二：本地任务恢复。** 对于父子关系的代理，系统支持通过进程内回调实现低延迟的结果传递。子代理完成后，其结果直接通过 `Promise.resolve()` 回传给父代理，无需经过文件系统中转。

双轨制的设计哲学是：**同步通信追求低延迟（本地任务恢复），异步通信追求高可靠（文件邮箱）**。代理可以根据通信的时效性需求选择合适的轨道。

## 9.8 领导者权限桥接

在协调者和 Swarm 模式中，领导者代理需要管理下属代理的权限。系统通过**权限桥接**（Permission Bridging）机制实现这一需求：

领导者的权限集合被计算为其自身权限与所有下属代理权限的**并集**。这意味着如果领导者有权执行某个操作，它可以将该权限"桥接"给下属代理。但桥接是单向的——下属代理无法通过桥接获得领导者本身不具备的权限。

```
领导者权限 = 自身基础权限 ∪ (下属代理₁权限 ∪ 下属代理₂权限 ∪ ...)
下属代理权限 ⊆ 领导者权限（子集约束）
```

权限桥接的另一个关键特性是**动态收缩**：当某个下属代理完成任务并被回收后，该代理贡献的权限也会从桥接集合中移除。这确保了权限集合始终与实际运行的代理集合保持一致，防止"权限泄漏"。
