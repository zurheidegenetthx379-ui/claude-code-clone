# 第六章：Skills 的技术实现细节与运行方式

## 6.1 三种技能来源概览

Claude Code 的技能系统采用三源架构，每种来源对应不同的发现路径、解析策略与安全模型：

| 来源类型 | 存储位置 | 发现时机 | 安全等级 |
|---------|---------|---------|---------|
| 文件型（File-based） | `.claude/skills/` 目录下 `.md` 文件 | 启动时扫描 + 文件变更监听 | 受沙箱保护 |
| 内置型（Built-in） | 编译进二进制的硬编码技能 | 启动时注册 | 最高信任 |
| MCP 型（MCP-sourced） | 通过 MCP 服务器动态提供 | MCP 握手后获取 | 需显式授权 |

文件型技能是用户最常接触的形态——只需在 `.claude/skills/` 目录放置一个符合格式的 Markdown 文件即可生效。内置型技能由开发团队维护，包含如 `/compact`、`/review` 等核心工作流。MCP 型技能则体现了 Claude Code 的开放哲学：外部工具服务器可以声明式地暴露新能力。

## 6.2 技能发现与解析流程

技能发现采用"扫描-解析-注册"三阶段流水线：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 目录扫描    │───>│ Frontmatter │───>│ 注册到      │
│ (glob递归)  │    │ 解析与校验  │    │ SkillManager│
└─────────────┘    └─────────────┘    └─────────────┘
```

**阶段一：目录扫描。** 系统使用 `glob` 模式递归遍历 `.claude/skills/**/*.md`，收集所有候选文件。扫描过程中会跳过隐藏目录和 `node_modules`，并使用 `fs.stat()` 获取每个文件的 inode 编号用于后续去重。

**阶段二：Frontmatter 解析。** 每个候选文件的头部被当作 YAML frontmatter 解析。解析失败不会导致整个扫描中断——系统会记录诊断日志并将该文件标记为 `invalid`，继续处理其余文件。这种容错设计确保了单个损坏的技能文件不会阻断整个技能系统。

**阶段三：注册与索引。** 解析成功的技能被注册到 `SkillManager` 的单例注册表中，以 `name` 字段为键建立索引。如果存在重名冲突，后发现的技能会覆盖先前的注册，并在日志中发出警告。

## 6.3 Frontmatter 字段全集

一个完整的技能文件 frontmatter 示例：

```yaml
---
name: "code-review"
description: "对指定文件执行代码审查"
paths: ["src/**/*.ts", "lib/**/*.js"]
model: "claude-sonnet-4-20250514"
effort: "high"
shell: "/bin/bash"
timeout: 30000
---
这里是技能的 Prompt 正文，使用 Markdown 编写...
```

各字段语义如下：

- **name**（必填）：技能的唯一标识符，用于命令触发和日志标记。必须为小写字母加连字符格式。
- **description**（必填）：自然语言描述，会被注入到系统提示词中，帮助 LLM 判断何时应调用此技能。
- **paths**（可选）：Glob 模式数组，当匹配路径的文件发生变更时自动触发技能。这是条件技能的核心。
- **model**（可选）：覆盖默认模型选择，允许为特定技能指定不同能力层级的模型。
- **effort**（可选）：推理努力级别（`low`/`medium`/`high`），影响 token 预算和响应深度。
- **shell**（可选）：覆盖默认 Shell 解释器路径，适配不同操作系统的脚本环境。
- **timeout**（可选）：执行超时时间（毫秒），防止技能长时间挂起。

## 6.4 内嵌 Shell 执行——最精妙的设计

技能系统中最具技术含量的设计是**内嵌 Shell 执行**能力。技能的 Prompt 正文中可以嵌入 Shell 代码块，这些代码块会在技能触发时被实际执行，输出结果被注入到最终的 LLM 上下文中：

```markdown
请先分析以下 git 日志，然后给出 commit 规范建议：

` ` `bash
git log --oneline -20
` ` `

` ` `bash
git diff --stat HEAD~5
` ` `
```

运行时，系统会：
1. 解析 Prompt 正文，提取所有 `` ```bash `` 代码块
2. 在受控 Shell 环境中依次执行
3. 将 stdout/stderr 捕获并替换回原代码块位置
4. 将展开后的完整文本作为 LLM 输入

这意味着技能不仅仅是静态的指令模板——它们是**活的程序**，能够感知当前项目状态并据此调整行为。

## 6.5 条件技能——paths 字段的 Hook 订阅模式

`paths` 字段实现了一种**声明式的事件订阅机制**。当一个技能声明了 `paths: ["src/**/*.ts"]`，它实际上是在说："当任何 TypeScript 源文件发生变更时，我应该被自动激活。"

实现上，系统在文件变更监听回调中维护一个反向索引：

```
变更文件路径 ──(glob匹配)──> 匹配的技能列表 ──> 触发执行
```

这种 Hook 订阅模式将"何时触发"的决策从代码逻辑中解耦到了声明式配置中，极大地降低了添加新自动化工作流的认知负担。

## 6.6 MCP 来源安全切断（RCE 防护）

MCP 型技能引入了一个关键的安全问题：外部服务器可能提供恶意的技能内容。系统采用了**多层切断机制**：

1. **能力声明校验**：MCP 服务器必须在握手阶段声明 `skills` 能力，未声明的服务器提供的技能会被静默丢弃。
2. **内容审查**：MCP 技能的内容会经过与文件型技能相同的 frontmatter 校验流程，格式不合规的内容不会注册。
3. **执行隔离**：MCP 技能的内嵌 Shell 执行会被限制在更严格的沙箱策略下运行，不允许访问 `.claude/` 配置目录。
4. **用户确认**：首次使用 MCP 技能时，系统会要求用户显式授权，防止"静默激活"攻击。

这套防护的核心思路是：即使 MCP 服务器被攻陷或恶意运营，攻击面也被限制在可控范围内。

## 6.7 inode 级去重

在多目录扫描场景下（例如用户同时拥有全局 `~/.claude/skills/` 和项目级 `.claude/skills/`），同一个技能文件可能通过符号链接出现在多个路径下。系统通过 `fs.stat().ino` 获取文件的 inode 编号进行去重：

```typescript
const seenInodes = new Set<number>();
for (const filePath of candidateFiles) {
  const stat = await fs.stat(filePath);
  if (seenInodes.has(stat.ino)) {
    continue; // 跳过已处理的同一物理文件
  }
  seenInodes.add(stat.ino);
  // ... 正常解析和注册
}
```

这种 inode 级去重避免了符号链接导致的重复注册问题，同时也优雅地处理了硬链接场景。相比基于路径字符串的去重方案，inode 方案更加鲁棒——它直接比较的是文件系统中的物理实体标识，而非逻辑路径。
