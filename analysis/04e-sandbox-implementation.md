# 第七章：Sandbox 技术实现细节与运行机制

## 7.1 四层执行链架构

Claude Code 的沙箱并非一个简单的"开关"，而是一条精心设计的四层执行链，每一层承担不同的职责：

```
用户请求 ──> [决策层] ──> [配置翻译层] ──> [权限集成层] ──> [执行与清理层]
               │              │                  │                 │
          是否使用沙箱    语义转换为平台参数   合并用户权限规则     启动容器/回收资源
```

**决策层**（`shouldUseSandbox()`）回答"要不要用沙箱"这个根本问题。**配置翻译层**（`convertToSandboxRuntimeConfig()`）将抽象的安全意图转换为具体的平台参数。**权限集成层**将用户配置的权限规则（allowlist/denylist）与沙箱默认策略合并。**执行与清理层**负责实际的容器生命周期管理。

## 7.2 shouldUseSandbox() 三维决策

`shouldUseSandbox()` 函数从三个维度评估是否启用沙箱：

**维度一：平台能力。** 沙箱依赖 Linux 内核的 namespace 和 cgroup 机制。系统首先检测当前运行平台是否支持这些底层原语。在 macOS 和 Windows 上，由于缺乏原生 namespace 支持，沙箱会优雅降级为基于进程权限的限制模式。

**维度二：操作风险评估。** 系统将待执行的操作分为三个风险等级：
- **只读操作**（`cat`, `ls`, `find`）：默认不启用沙箱，以减少延迟开销
- **写入操作**（文件编辑、创建）：根据配置决定是否启用
- **高危操作**（Shell 命令执行、网络请求）：强制启用沙箱

**维度三：用户配置。** 用户可以通过 `settings.json` 中的 `sandbox` 字段显式覆盖自动决策结果。支持 `"always"`、`"never"` 和 `"auto"` 三种模式。

三个维度的评估结果通过"最严格优先"策略合并——任何一个维度要求启用沙箱，最终决策就是启用。

## 7.3 convertToSandboxRuntimeConfig() 语义翻译

`convertToSandboxRuntimeConfig()` 是架构中最容易被低估但实际至关重要的组件。它的职责是将 Claude Code 内部的**抽象安全语义**翻译为具体沙箱实现能理解的**运行时参数**。

这种翻译存在两种路径语义的差异：

### 7.3.1 路径语义差异

**内部语义（Claude Code 视角）**：使用逻辑路径描述，如 `"project-root"`, `"user-config-dir"`, `"tmp"`。这些路径是相对于项目上下文的抽象概念。

**运行时语义（沙箱实现视角）**：需要绝对路径、挂载选项（`read-only`/`read-write`）和设备权限（`/dev/null`, `/dev/urandom`）。

翻译函数维护一张映射表：

```typescript
const pathMapping: Record<LogicalPath, SandboxMountConfig> = {
  "project-root":  { hostPath: resolveProjectRoot(), containerPath: "/workspace", mode: "rw" },
  "user-config":   { hostPath: "~/.claude",          containerPath: "/config",    mode: "ro" },
  "tmp":           { hostPath: os.tmpdir(),           containerPath: "/tmp",       mode: "rw" },
};
```

## 7.4 Git 裸仓库逃逸防护——两阶段防御

沙箱面临的一个隐蔽攻击向量是**Git 裸仓库逃逸**。攻击者可以构造一个包含 `.git` 裸仓库的项目目录，通过 `GIT_DIR` 环境变量或 `--git-dir` 参数，诱导沙箱内的 Git 操作读写沙箱之外的文件系统区域。

系统实施了**两阶段防御**：

**阶段一：环境净化。** 在沙箱启动前，系统会扫描并清除所有 Git 相关的环境变量（`GIT_DIR`, `GIT_WORK_TREE`, `GIT_INDEX_FILE` 等），确保沙箱内的 Git 进程不会受到外部注入的仓库路径影响。

**阶段二：`.git` 目录验证。** 系统检查项目目录中的 `.git` 是文件还是目录。如果是文件（gitlink），系统会解析其内容，验证指向的裸仓库路径是否在沙箱允许访问的挂载列表内。如果指向外部路径，系统会拒绝在该目录中执行 Git 操作。

## 7.5 设置文件与技能目录保护

沙箱对以下目录实施特殊的保护策略：

- **`~/.claude/settings.json`**：以只读方式挂载。沙箱内的进程可以读取配置，但无法篡改设置文件来放宽安全策略。
- **`.claude/skills/`**：以只读方式挂载。防止沙箱内的代码通过创建恶意技能文件来获得后续执行的持久化能力。
- **`~/.claude/credentials`**：完全不挂载进沙箱。沙箱内进程无法访问用户的 API 密钥和认证令牌。

这种分层保护确保了即使沙箱内的代码存在恶意行为，也无法突破沙箱边界修改影响系统行为的持久化配置。

## 7.6 autoAllow 不是无条件放行——三重条件

系统中存在一个常被误解的机制：`autoAllow`。名称暗示"自动允许"，但实际上它是一个**有条件自动允许**机制，需要同时满足三重条件：

1. **操作在白名单中**：只有预先声明为安全的操作类别（如读取项目文件、执行已注册的 MCP 工具）才可能被自动允许。
2. **无 Hook 拦截**：如果存在任何 beforeToolUse Hook 且返回了 `deny`，即使操作在白名单中也会被拒绝。
3. **会话状态正常**：如果当前会话处于受限模式（如 token 用量接近限额），自动允许会被临时挂起。

三重条件的"与"逻辑确保了 `autoAllow` 永远不会成为安全漏洞的入口。这也是为什么代码中将其命名为 `autoAllow` 而非 `forceAllow`——它始终保留被否决的可能。

## 7.7 热重载与诊断

沙箱配置支持**热重载**：当 `settings.json` 被修改时，系统会通过文件监听器检测到变更，在不中断当前执行的情况下，将新的沙箱策略应用到后续操作中。

热重载的实现依赖一个**版本化策略栈**：每次配置变更会 push 一个新版本的策略对象到栈中，新的操作从栈顶获取最新策略，而已经在执行中的操作继续使用创建时的策略版本。这避免了运行时策略变更导致的一致性问题。

**诊断模式**可以通过 `--sandbox-debug` 标志启用，该模式会输出：
- 每次 `shouldUseSandbox()` 调用的三维评估详情
- 配置翻译前后的对比 diff
- 权限合并的逐步 trace
- 沙箱启动的完整参数 JSON

这些诊断信息对于排查"为什么某个操作被沙箱拒绝"这类问题至关重要。
